The Problem

Indian students navigating competitive exams operate across 5+ disconnected platforms — exam alerts on one, college rankings on another, tutoring on a third, career guidance elsewhere. No unified account. No cross-module intelligence. Every platform treats them as a new user with no history.

Why Existing Solutions Failed

CollegeDunia, Shiksha and similar platforms provide information but not tools. No rank prediction engine, no OTP-protected booking, no integrated workflow. They are content sites, not operating systems.

Architecture

Single Next.js codebase with shared MongoDB instance. Unified user schema across all four modules. OTP authentication via crypto.randomInt() — chosen over Firebase Auth to reduce latency on Indian mobile networks. MSG91 for WhatsApp OTP, Resend for transactional email. Middleware-based route protection on all API routes.

Key Engineering Decision

Chose a single Next.js monolith over a monorepo or microservices. Deployment stays atomic, inter-service latency is zero, shared auth and DB connections are trivial. The solo constraint made this the correct call — operational simplicity compounds when you are the only engineer.

The Problem

Enterprise asset tracking at scale fails with paper logbooks — no conflict detection, no audit trail, no visibility into who has what or when. Double-bookings are discovered only after the damage is done, often mid-operation.

Architecture

Normalized schema separating assets, reservations, and users. Date-window queries use compound indexes on (assetId, startDate, endDate) to detect overlapping reservations in O(log n). Atomic updates via MongoDB transactions prevent TOCTOU race conditions. QR codes map to asset+reservation records for instant scan-to-log workflows.

Key Engineering Decision

Approval logic is serializable — no two approvals for the same asset window can succeed simultaneously. The tradeoff is slightly higher write latency, acceptable for a booking system where correctness is non-negotiable.

Campus information lives across 10+ disconnected systems. IITR Nexus routes a single query across academics, library, dining, events, and notices using multiple MCP servers in parallel. Intent classification determines which servers to query. Confidence scoring ensures grounded responses — if the system is uncertain, it surfaces the source, not a guess. The Problem

IIT Roorkee students navigate 10+ disconnected portals for academics, library, dining, events, and notices. Simple questions like "is the library open?" require navigating separate systems with separate logins. The fragmentation is the problem, not the information itself.

Architecture

Multiple MCP servers each owning one domain. Intent classifier routes the query to relevant servers. Parallel retrieval fetches simultaneously. Synthesis layer merges results and computes confidence. When confidence is below threshold, response surfaces the source URL instead of generating an answer — explicit hallucination resistance.

Key Engineering Decision

Parallel retrieval over sequential routing. Sequential would add latency proportional to number of servers queried. Parallel keeps response time bounded regardless — critical for a chat interface where latency directly affects perceived quality.

Canopy parses Python and JS/TS import statements directly from GitHub — no cloning, no local execution, no CI setup. Builds an interactive React Flow dependency graph with Gemini-generated module summaries. MD5 caching prevents redundant parses. Dark forest UI for extended analysis sessions.

The Problem

Onboarding to a new codebase takes days. Existing tools require cloning locally, running scanners, generating static output. No instant, shareable, interactive view of what depends on what.

Why Existing Solutions Failed

Tools like Madge or dependency-cruiser are local-only, produce static SVG, require the full repo on disk. Unsuitable for PR analysis or cross-team onboarding. Canopy works entirely from a GitHub URL — no setup, no install, shareable link.

Architecture

GitHub API fetches file trees without cloning. AST-based import parser handles Python and JS/TS. MD5 hash of file content as cache key — parsed results reused when content has not changed. React Flow renders interactively. Gemini generates one-sentence summaries per module from docstrings and imports.

The Problem

Candidate ghosting is a coordination failure. Recruiters lose context between pipeline stages, ATS systems do not surface candidate-facing timelines, manual updates do not scale. The candidate experience breaks not because companies do not care, but because the tooling does not surface the right action at the right moment.

Architecture

ATS webhook → Signal ingestion API → event classification → AI template selection and personalization → candidate update dispatch. SLA engine on a background job monitors stage-to-stage elapsed time against configurable thresholds. Breached SLAs surface in recruiter dashboard. Candidate timeline is public-facing, no login required.

Key Engineering Decision

Active rather than passive: Signal reaches out when something changes, with context derived directly from the ATS event that triggered it — not a generic drip campaign that fires on a schedule regardless of what has happened.

Dark Patterns Spotted

Deliberate design decisions made against users to protect business metrics. All from personal experience with Indian apps.

Roach Motel
Easy in. Impossible out.

Signup takes 30 seconds. Cancelling takes 7 screens, a call, and a waiting period. The friction is intentionally asymmetric — exactly what I experienced with Paytm's refund flow.

spotted in: Paytm refunds · Indian telecom portals · OTT cancellations
Unverifiable Refund
Rs.1,600 displayed. Rs.600 processed.

MakeMyTrip showed a refund amount at the start of the cancellation flow. The actual amount processed was Rs.1,000 less with no itemised breakdown. You only discover the gap after completing cancellation — too late to back out.

spotted in: MakeMyTrip cancellation · 4-step IVR designed for drop-off · AI agent deployed as human
Hidden Costs
The price that only appears at the last step.

Four screens of checkout. Everything looks fine. Final screen: platform fee Rs.5, convenience charge Rs.12, GST Rs.18. By step 4 you are already committed — and they know it.

spotted in: BookMyShow · MakeMyTrip · every Indian ticketing platform
Fake Scarcity
Countdown timers that reset. "Only 2 left" that never runs out.

Refresh after 10 minutes — still 3 rooms, timer reset. Manufactured scarcity to trigger loss aversion. You are not going to miss out. They just need you to feel like you might.

spotted in: MakeMyTrip · Goibibo · Flipkart sale pages
Confirmshaming
Making the "no" feel like shame.

One button: "Yes, continue." The other: "No thanks, I don't want to save money." The choice is real. The framing is manipulation.

spotted in: subscription apps · food delivery premium plans · e-commerce flash sales
Not all friction is a dark pattern. Paytm's cancellation flow is one — designed to make you give up. Zomato's cancellation warning is borderline — it informs you of a real consequence. The difference is intent: does the friction protect the user or exploit them?
